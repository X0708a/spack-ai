# Spack-AI Diagnostic Bridge

Spack-AI Diagnostic Bridge is organized as a small production-style research
prototype for a GSoC-style qualification task: the original evaluation pipeline
lives under `eval/`, the CI-aware extension lives under `ci_extension/`, and
shared logic sits in `shared/`.

At its core, the project is a multi-stage pipeline that extracts Spack package
metadata, compresses it into a token-aware JSON summary, feeds that summary to a
local or cloud LLM to generate Off-Leading-Edge (OLE) risk scenarios, and then
extends those scenarios into scoring, scheduling, simulated CI, and
feedback-driven reprioritization.

## Quick Start

```bash
pip install -r requirements.txt

# Step 1 — extract metadata
python3 eval/summarize.py --force-static        # no Spack needed
python3 eval/summarize.py                       # uses live Spack if installed

# Step 2a — run with LM Studio (local, no API key)
python3 eval/generate_scenarios.py \
  --lm-studio \
  --lm-studio-url "http://localhost:1234/v1" \
  --lm-studio-model "qwen2.5-coder-7b-instruct-mlx" \
  --output-json eval/scenarios.json

# Step 2b — run with Anthropic API
export ANTHROPIC_API_KEY=sk-ant-...
python3 eval/generate_scenarios.py --output-json eval/scenarios.json

# Step 2c — deterministic fallback (no API, no LM Studio)
python3 eval/generate_scenarios.py --mock-only --output-json eval/scenarios.json

# Suppress already-validated CI specs
python3 eval/generate_scenarios.py \
  --validated-spec "spack install root@6.30.06 ^libxml2@2.13.0 ^clhep@2.4.7.1 +python +roofit"

# Run the CI-aware extension end to end
python3 ci_extension/main.py --mock-only --deterministic --max-tests 5 --force-static
```

## LLM-Backed Scenario Generation

Scenarios are generated by feeding the compressed `eval/summary.json` into an
LLM using the system prompt defined in `eval/generate_scenarios.py`. The
summarizer keeps the payload under the token budget and focuses the prompt on
recent versions, guarded variants, and unbounded dependencies, which are the
primary drivers of OLE risk.

The script supports three backends:

- Anthropic API
- local LM Studio
- deterministic fallback

All three backends share the same validation, deduplication, and reporting path,
so the downstream behavior stays consistent regardless of how candidate
scenarios were produced.

### What the pipeline did with this output

- Parsed the JSON payload successfully even though the model returned a wrapper object.
- Dropped scenario 3 (`geant4@10.7.4`) because it was too similar to `geant4@11.0.4` with `distance=0.170 < threshold=0.180`.
- Topped up from the deterministic fallback pool to reach the required count of 3 distinct scenarios.
- Emitted final outputs with `source: lm-studio` in `eval/scenarios.json` and `eval/spack_spec.md`.

### What the model got right

- Middle-aged primary versions: `root@6.28.12` (`vs[2]`), `geant4@11.0.4` (`vs[2]`)
- Correct OLE direction: old primary plus newest available dependency version
- Correct `risk_vector` format: `dep@version`

### Known limitation of the 7B model on this prompt

The dependency was occasionally written without the `^` caret prefix
(`clhep@2.4.7.1` instead of `^clhep@2.4.7.1`). The structured `risk_vector`
field was still correct, so the pipeline could validate and retain the
scenario. Using `--small-model` swaps in a simplified prompt with explicit
WRONG/RIGHT syntax examples and largely eliminates this issue on smaller local
models.

## CI-Aware Extension (My Contribution)

The CI-aware extension keeps the original evaluation pipeline intact and layers
production-oriented decision-making on top of it.

- `ci_extension/scoring.py` ranks scenarios using unbounded dependency count,
  version distance from the newest release, rarity against prior runs, and
  nearby failure history.
- `ci_extension/scheduler.py` selects the highest-value scenarios for the next
  CI run, filters duplicate fingerprints, excludes already validated specs, and
  batches tests with overlapping dependency graphs.
- `ci_extension/ci_runner.py` simulates execution in mock or deterministic mode
  and can optionally perform real `spack spec` checks.
- `ci_extension/feedback.py` updates failure history, updates the validated spec
  cache, and boosts future scores for scenarios that are similar to prior failures.

### Extension Pipeline

```text
summarize -> generate -> deduplicate -> score -> schedule -> test -> feedback
```

## Deduplication

`eval/generate_scenarios.py` uses a weighted distance metric over dependency
sets, dependency versions, primary package versions, and variants so that near-
identical OLE scenarios do not crowd out the final report.

### Deduplication Demo

```bash
python3 eval/generate_scenarios.py --mock-only --demo-dedup
```

```text
-- Deduplication demo --
[warn] deduplicated scenario 'spack install root@6.30.07 ...' (distance=0.003, threshold=0.180)
input=4 output=3 threshold=0.18

-- Pairwise distances --
scenario 1 vs 2: 1.000
scenario 1 vs 3: 0.455
scenario 2 vs 3: 1.000
```

### Distance Weights

| Component | Weight | What it measures |
|---|---:|---|
| dep-set Jaccard | 0.40 | which dependencies are present |
| dep-version distance | 0.25 | same dependency, different pinned version |
| primary version distance | 0.20 | how far apart the primary releases are |
| variant set Jaccard | 0.15 | enabled and disabled feature flags |

Previously validated CI specs are passed via `--validated-spec` and treated as
an exclusion list. Each scenario also carries a SHA-1 fingerprint so that cache
lookups and duplicate suppression stay cheap across repeated runs.

## Model Compatibility Notes

The prompt design works across both local and hosted models, but quality varies
with model size and prompt-following strength.

| Model | Size | Flag needed | Quality |
|---|---:|---|---|
| `qwen2.5-coder-7b-instruct-mlx` | ~4.5 GB | none | Good — used for this submission |
| `qwen2.5-coder-3b-instruct` | ~2.0 GB | `--small-model` | Acceptable |
| `mistral-7b-instruct-v0.3 Q4_K_M` | ~4.1 GB | none | Best for 8 GB machines |
| Anthropic `claude-sonnet-4` | API | none | Highest quality |

The `--small-model` flag swaps in a simplified prompt with explicit WRONG/RIGHT
examples for Spack syntax and OLE direction. It is designed for models under
7B that struggle with dense abstract instructions but can follow concrete
pattern-matching examples reliably.

## Exit Codes

| Code | Meaning |
|---|---|
| `0` | success |
| `1` | bad input such as missing or malformed `summary.json` |
| `2` | could not assemble 3 distinct scenarios after deduplication |
| `3` | `anthropic` package is missing; install it with `pip install anthropic` |
